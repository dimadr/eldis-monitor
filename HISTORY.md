## Этап 5: Исправления и стабилизация (Май 2026)

### 5.1 Проблема: пустые отчёты

**Симптом:** Пользователь вводит ID Элдиса (например, `dd087e83-a704-417f-a424-50aeef9fb3b6`), система получает метаданные (адрес, прибор, серийный номер), но отчёт показывает «За указанный период данных нет» или «Все показания в норме. Отклонений не выявлено.»

При этом на сайте ЭЛДИС (eldis24.ru) и в старых отчётах за тот же период данные есть — полный комплект: температуры, объёмы, масса, тепловая энергия за каждый день.

**Диагностика:**
1. Тест API напрямую: `deviceID` → `hotWater: []` (пусто)
2. Тест с объектным ID (`objectID`) → тоже пусто
3. Тест с каждым отдельным ID ввода из `2191.csv`:
   - `8a8dd1af-b3af-454d-9ec4-a5ba3fa81361` (ТВ1, ТС) → **10 записей данных** ✅
   - `dd087e83-a704-417f-a424-50aeef9fb3b6` (ТВ2, ГВС/ХВС) → пусто ❌
4. Вывод: **API ЭЛДИС возвращает данные только для конкретного ввода**, а не для прибора или объекта целиком. Каждый input ID — отдельный канал данных.

**Корневая причина:** Старая логика `generate()` опрашивала только один ID — тот, что передал пользователь. Если пользователь передавал ID ввода без данных (ТВ2), отчёт оставался пустым, даже если другие вводы того же прибора (ТВ1) содержали полные данные.

### 5.2 Решение: сбор данных со всех вводов

**Шаг 1 — База маппинга (`mapping.db`):**
- Получен файл `2191.csv` с флешки пользователя (SanDisk Ultra Fit 14.3GB, exfat)
- CSV содержит 2192 записи со столбцами: `id` (ID Элдис), `objectID`, `deviceID`, `address`, `deviceName`, `deviceCode`, `sn`, `resourceName`, `measurePointName`, `modemSN`, `modemModelName`, `status`, `createdOn`, `isArchivalRecord`
- Создана SQLite база `/root/eldis/mapping.db` с индексами по `id`, `objectID`, `deviceID`, `sn`, `modemSN`, `address`
- Скрипты: `init_mapping_db.py` (создание/пересборка), `query_mapping.py` (поиск по любому полю: `eld_id`, `obj_id`, `dev_id`, `sn`, `modem`, `addr`, `device`)

**Пример маппинга для объекта «ТСЖ "Элит", Ульяновская ул, д. 15»:**
```
            Eldis ID (id)                      | Ресурс | Точка | SN
8a8dd1af-b3af-454d-9ec4-a5ba3fa81361         | ТС     | ТВ1   | 186420
f6818a78-2faf-4d52-ae9d-d93e5974a6bd         | ГВС,ХВС| ТВ2   | 186420
dd087e83-a704-417f-a424-50aeef9fb3b6         | ГВС,ХВС| ТВ2   | 186420
```
Все три ввода принадлежат одному deviceID: `bbe6337a-6235-4f5a-b5bc-262a12131477` (ВКТ-7, SN: 186420).

**Шаг 2 — Функция `get_all_inputs_for_device()`:**
- Принимает любой Eldis ID
- Ищет его в `mapping.db`
- Находит `deviceID` этого ввода
- Возвращает список ВСЕХ вводов с тем же `deviceID`

**Шаг 3 — Изменение `generate()`:**
- Вместо одного `get_readings(device_id, ...)` теперь:
  1. Получает список всех вводов через `get_all_inputs_for_device()`
  2. Для каждого ввода вызывает `get_readings(inp_id, ...)` отдельно
  3. Все результаты объединяются в общий `normalized` массив
  4. `process_readings()` обрабатывает объединённые данные
- Fallback: если маппинг не найден (нет файла или ID не в базе), работает как раньше — опрашивает только переданный ID

**Результат:** Пользователь вводит любой ID (хоть ТВ1, хоть ТВ2) — отчёт содержит данные со **всех** вводов прибора. Для объекта «ТСЖ "Элит"» — 31 запись за март 2026 (t1, t2, dt, V1, V2, M1, M2, Q, P1, P2, QntHIP, VOCR).

**Проверка:**
```bash

curl http://localhost:5000/generate -d "device_id=dd087e83...&start_date=2026-03-01&end_date=2026-03-31"
# Результат: "За указанный период данных нет."

# После исправления — 31 запись
curl http://localhost:5000/generate -d "device_id=dd087e83...&start_date=2026-03-01&end_date=2026-03-31"
# Результат: Таблица 1 с 31 строкой, Средние, Итого
```

### 5.3 Исправленные баги в server.py

**1. `ELDISS_API_KEY` — опечатка (критическая)**
- Строка 465: `headers={"key": ELDISS_API_KEY}` — двойная `S`
- Должно быть: `ELDIS_API_KEY` (константа определена на строке 28)
- Результат: при выполнении этого блока кода — `NameError`, падение функции `generate()`
- Исправлено: `ELDISS_API_KEY` → `ELDIS_API_KEY`

**2. Небаланс масс — сломанная логика `if/else` (критическая)**
- `perform_checks()`: блок проверки небаланса масс (строки 706–731) содержал ошибку привязки `else`
- ```python
  if M1_avg > 0:           # ← внешнее условие
      imbalance = ...
      if -2 <= imbalance <= 2:  # ← это должно проверяться
          checks.append(...)    # ok
      # ← здесь не хватало else для ошибки!
  else:                    # ← это else привязано к if M1_avg > 0, а не к if imbalance!
      checks.append(...)   # ← ошибка, добавляется когда M1_avg <= 0
  ```
- Последствия:
  - Если `M1_avg > 0` и `imbalance` в пределах ±2% → добавляется `ok` ✅
  - Если `M1_avg > 0` и `imbalance` ВНЕ ±2% → **проверка НЕ ДОБАВЛЯЕТСЯ ВООБЩЕ** (молча пропадает) ❌
  - Если `M1_avg <= 0` → добавляется `error` с `imbalance` = неопределённая переменная → **NameError** ❌
- Исправлено: блок переписан, используется единая переменная `status` и условное добавление `description`

**3. Температура 0°C — falsy check**
- `perform_checks()` (строка 807): `if r.get('t1') and r.get('t2')`
- В Python: `0` — falsy. Если датчик показывает 0°C (реальная ситуация для холодной воды зимой), строка игнорируется
- Исправлено: `if r.get('t1') is not None and r.get('t2') is not None`

**4. Почасовое агрегирование — M1 и M2 из dM**
- `get_readings()` (строки 186–187): оба поля `M1` и `M2` получали одно и то же значение `dM`
- То же для `V1`/`V2` — оба получали `dV`
- Последствие: `M1 == M2` всегда → `imbalance = 0%` → проверка небаланса всегда показывает `ok`, скрывая реальные проблемы
- Исправлено: используются раздельные поля `M1`, `M2`, `V1`, `V2`

**5. Поиск ObjectID — инвертированное условие**
- `generate()` (строка 461): `if len(device_id) > 36 or '-' not in device_id[:36]`
- UUID имеет длину 36 и содержит дефисы. Условие:
  - Для UUID (36 символов, с дефисами): `36 > 36 → False`, `'-' not in "550e..." → False` → **не входит в блок**, хотя это как раз те ID, для которых нужен поиск objectID
  - Для коротких ID (без дефисов, < 36): `True` → **входит в блок** без необходимости
- Исправлено: `if '-' in device_id and len(device_id) == 36`

**6. Дублирование строк**
- `generate()` (строки 614–615): дважды подряд `filtered = readings_table`
- Просто удалена строка 614

**7. Жёстко прописанные пути**
- `ns_dictionary.json`: `"/root/eldis/eldis-monitor/ns_dictionary.json"`
- `mapping.db`: `'/root/eldis/mapping.db'`
- Проблема: при переносе проекта на другой сервер пути сломаются
- Исправлено: используется `os.path.join(BASE_DIR, ...)` для относительных путей

**8. Дублирование импортов**
- `import re` — многократно внутри функций (строки 69, 649, 654, 659, 846)
- `from datetime import datetime` — внутри функций (строки 112, 148, 173, 320)
- `from collections import defaultdict` — внутри функции (строка 167)
- Не критично, но замедляет выполнение и засоряет код

**9. Небаланс масс — NameError при отсутствии данных по теплу**
- После отката на версию 1.87 исправление пункта 2 было потеряно
- При формировании отчёта для приборов без данных по отоплению (только ГВС/ХВС) `perform_checks()` падала с `NameError: cannot access local variable 'imbalance'`
- Причина: `else:` был привязан к `if heat_readings:`, а не к `if imbalance`. При пустых `heat_readings` переменная `imbalance` никогда не присваивалась
- Исправлено повторно 20.05.2026: блок полностью переписан — добавлены отдельные обработчики для всех трёх вариантов (нет данных, M1_avg <= 0, M1_avg > 0 с проверкой диапазона)
